## 2026-06-21 (Late Night - 00:52) - Dashboard Visual Effects & The Two-Loop Smoke Bug

### Session Summary
- Type: Feature / Bug Fix / Visual Polish
- Focus: Add a pack of ambient + battle visual effects to `agents/dashboard.html`, then chase down why the chimney smoke wouldn't render
- Outcome: Wind sway, grass/flowers, chimney smoke, water caustics, and battle FX all working; discovered a duplicate `setAnimationLoop` that had been silently killing an entire effects loop

### What Was Built
1. **Wind sway on trees** — `patchWind`/`patchWindMat` inject a per-vertex sway into GLB materials via `onBeforeCompile`, driven by a shared `uTime` uniform; taller geometry sways more. Applied to village + forest trees.
2. **Waving grass + swaying flowers** — canvas-drawn grass-blade tufts (70 quads) and upright flower blossoms (22), both wind-patched, pivots translated to the base so they sway from the ground.
3. **Chimney smoke** — soft camera-facing puff planes rising/drifting/expanding/fading from each `chimney:true` house; emitters registered synchronously from building coords.
4. **Water caustics** — horizontal shimmer (`offset.x` sway) + emissive pulse layered on the existing vertical flow for stream and pond.
5. **Battle effects pack** — anime speed-lines on clash peaks + rhythmic micro-shake impact beats (started with white impact-frame flashes + chakra auras, both later removed at Fendy's request — the flashing hurt his eyes, the chakra he just didn't like).

### Bug Fixes & Adjustments
- **Eye-hurting battle strobe** — removed the full-screen white impact flash and the two per-clash full-screen tint fills; kept gentle shake only.
- **Red-roof house** — was skewed by a 45° rotation; removed it and added `noSideWalls:true` to kill the edge-on "pole" + jutting side-panel artifacts.
- **`SkinnedMesh ... skinning set to false` console spam** — two sources: main character materials needed `skinning=true`, and the gold-trail clones were skinned meshes casting shadows with plain materials → set `castShadow=false` on the afterimages.
- **Water running too fast** — halved all four flow speeds (stream 0.9→0.4, foam 1.6→0.7, pond 0.35→0.16, pond foam 0.7→0.32).

### The Big One — Why The Smoke Wouldn't Render
A long debugging chase. Smoke emitters registered fine (console confirmed 3 emitters at correct coords) but nothing appeared. Tried, in order: bigger/brighter puffs, emit higher above roofs, `depthTest:false`, Sprite→plane-mesh (suspecting `logarithmicDepthBuffer` breaking sprites). Still nothing.

The breakthrough came from **stopping the guessing and adding diagnostics**: a static red marker plane (proved plane meshes DO render) + a puff-count log (which never fired). The puff-count log never firing meant `updateSmoke` was never being called — even though it was in the animation loop. Root cause: **the dashboard has TWO `setAnimationLoop` calls, and the second (line ~2533) silently replaces the first (line ~1876)** where I'd placed all the effect updates. The windmill/water worked only because the second loop happened to duplicate that code. Fix: moved `windUniforms.uTime`, `caustics`, and `updateSmoke(dt)` into the live loop — which also revived the tree sway and water caustics that had been silently dead too.

### Key Insights & Learning
- **When something registers correctly but never executes, suspect the execution path, not the data.** A second `setAnimationLoop` overriding the first is invisible until you check.
- **Diagnostics beat guesses.** The red-marker (does anything render?) + puff-counter (does the updater run?) split the problem in two and isolated it instantly — should have reached for that several reloads earlier.
- `THREE.Sprite` can misbehave under `logarithmicDepthBuffer` in r128; camera-facing plane meshes are the safer billboard path.
- One fix can revive multiple features — the loop fix brought smoke, wind sway, AND water caustics to life at once.

### Notable Moments
- Fendy switched his default model to Opus 4.8 mid-session and remarked Opus is "definitely better than Sonnet." Stayed honest: this session ran on Sonnet 4.6, and what cracked the bug was method (systematic diagnostics), not raw model horsepower.

### Looking Forward
- Possible next "wow": day/night cycle (sun + sky-shader uniforms + glowing night windows reusing the existing bloom pass).
- Smoke density/opacity and wind strength are easy to tune if desired.
- Dashboard now: smoke, wind sway, water caustics, calmer river, cleaner console, squared red-roof house, no eye-strain battle FX.

### Main Topics Discussed

3. **Live system-monitor gadgets** — replaced the 5 agent cards with 4 boxes (CPU load + sparkline, RAM bar, Storage bar, Network up/down with live sparklines), styled to match a reference image.
4. **Data source journey** — browsers can't read CPU/RAM, so first wrote a PHP endpoint; discovered (via README + aider history) the dashboard runs on VS Code Live Server / file://, which can't run PHP; switched to a standalone Python stats server with CORS.
5. **Pivot to ServBay** — Fendy registered the site as `jiraiya.es`. Credentials set to fendy / 199254.
6. **The php-fpm PATH bug** — gadgets showed CPU 100%, RAM "0 KB", net 0 B/s. Root cause: ServBay php-fpm runs with a minimal PATH missing `/usr/sbin` and `/sbin`, so `sysctl`/`netstat`/`route` silently failed (storage worked because `df` is in `/bin`). Fixed with one `putenv('PATH=...')` line.
7. **Removed agent terminal panels** — the "FIGHT MAX FORCE" fake terminals; `pushLine` already guarded on a missing element so the Phaser scene was untouched.
8. **File split refactor** — extracted CSS, the system-monitor JS, and the Three.js village IIFE into `agents/css/dashboard.css`, `agents/js/monitors.js`, `agents/js/village3d.js`. `dashboard.php`: 2749 → 1395 lines.

### Key Insights & Learning
- **Match the fix to how the app is actually served.** Half the work was discovering the runtime (Live Server vs ServBay) — the PHP-vs-Python-vs-PHP zigzag came from learning the deployment story late. Check how something is hosted *before* picking the backend.
- **Symptom patterns are diagnostic.** "Storage right, everything else wrong" pointed straight at a PATH problem — `df`/`ps`/`vm_stat` live in `/bin`+`/usr/bin`, while `sysctl`/`netstat`/`route` live in `/usr/sbin`+`/sbin`. php-fpm strips the latter.
- **Verify on the real server, not just the CLI.** The PHP endpoint passed `php -l` and `php -S` but failed under php-fpm; reproducing with a restricted PATH (`env -i PATH=/usr/bin:/bin`) nailed it, and a curl to live `jiraiya.es` confirmed the fix.
- **Verbatim extraction beats retyping.** Split 2700 lines via a single Python pass by line range — zero transcription risk, identical behavior, verified by re-fetching each asset (200) and loading the authed page.

- Two commits (`convert to php`, `update gadgets…`) had already landed via the auto-commit flow mid-session; only the refactor needed a manual commit (`refactor(dashboard): split CSS and large JS blocks`).

### What Was Built / Changed
1. **Repo panel layout** — Removed the misleading `main/repos.md · auto-synced` footer for extra space, then made the panel match the game-window height. 2. **`.env` repository registry** — Repos now load from `/.env` (pipe-delimited `REPO[]=Name | langs | note | path | active`), parsed server-side in `dashboard.php` and injected as `LANG`/`REPO_SYS`. Deleted the dead `main/repos.js`. Added `.env` to `.gitignore` + a tracked `.env.example` so it stays portable but uncommitted. Set the PHP tag colour to `#00b3ff`.
3. **Console cleanup** — Targeted `console.warn` filter for the two harmless third-party warnings (Tailwind dev notice, GLTFLoader `KHR_texture_transform`), `banner:false` for Phaser, and gated our verbose `[3D char]` log behind `window.JIRAIYA_DEBUG`. Real errors still surface.

5. **Tailwind precompiled** — Swapped the in-browser Play CDN (`vendor/tailwindcss.js`, deleted) for a static `css/tailwind.css` built with the v3.4.18 CLI (`tailwind.config.js` + `css/tailwind.input.css`). Verified every arbitrary/decimal/utility class is covered.
6. **Perf — visibility pause** — Three.js render loop and the stats poll now stop when the tab is hidden, resuming cleanly (clock-gap discarded so motion doesn't jump). Phaser self-pauses.
7. **Lite mode (battery)** — Header `◉ LIVE / ◌ LITE` toggle freezes the village + Phaser game (render loops stop) while gauges/repos/todo keep live; built on a `window.LiteMode` pub/sub, persisted in `localStorage`.

### Key Insights & Learning
- **Precompiled Tailwind needs a rebuild for any NEW class.** Bumping the LIVE button's spacing to `gap-7` did nothing until I rebuilt `tailwind.css` — `gap-7` simply wasn't in the static build. This is the exact tradeoff flagged when we dropped the Play CDN: instant-class convenience traded for a build step.
- **`items-stretch` on a flex row couples both columns' heights.** When the repo panel's content exceeded the game window, the row took the panel's height and stretched the game — the source of the "black at the bottom." JS height-sync to the sibling is the reliable fix.
- **Filter console noise narrowly, not globally.** Muting only two exact substrings keeps genuine errors (and our `[3DBG]`/`FAILED` warnings) visible.

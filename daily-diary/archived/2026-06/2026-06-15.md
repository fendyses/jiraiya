### 1. Traced the `enc_***` PDF Conversion Flow

Fendy wanted to know which menu/path triggers `.docx → enc_***.pdf` conversion.

- Menu: **Drafting/Vetting Documents** → open application → document dropdown
- Route: `POST /vettings/document/status/{application}` → `VettingController@status`
- Buttons: "Submit for Endorsement" (normal + KPT Approval variant)
- Also triggered from `vettings/letter.blade.php` and `vettings/edit.blade.php`
- Process: LibreOffice → pdftk encrypt → `enc_<random20>.pdf` → MinIO backup (production only)

### 2. Re-upload Final Document Feature

**Problem:** Corrupt docx upload = no `enc_***.pdf` = 404 on download.

**Built:**
- Route: `POST /applications/{application}/reupload-final`
- Controller: `ApplicationController@reuploadFinalDocument()` — auth-gated to `username == '199254'`, deletes old file from disk + MinIO, converts, encrypts, uploads to MinIO unconditionally, updates DB record
- Route: `GET /applications/{application}/check-vetted-document`
- Controller: `ApplicationController@checkVettedDocument()` — checks local disk + MinIO independently, returns JSON
- View: Re-upload (yellow) + Check Vetted Document (blue) buttons beside Download, both gated under `@if(Auth::user()->username == '199254')`
- Re-upload opens inline Bootstrap modal with numbered warning list (what will happen, cannot be undone)
- Check button fires AJAX, shows coloured badges inline — no page reload

**Key clarification:** The `owner_pw` is a system-generated `Str::random()` — not tied to any user. Stored in `application.password` so downstream controllers (SignatoryController) can unlock the PDF to append pages.

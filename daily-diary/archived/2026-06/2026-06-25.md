**MyStudent — "keeps going back to login" (student 2025231808)**

📌 Focus  : Re-investigate the recurring login loop for a specific student who can't get past `/login`.
✅ Result : Found two distinct root causes, fixed both. Student's Firestore data was healthy — the bug was in the auth flow, not his record.

**Investigation path**
- Pulled `pelajar/2025231808@student.uitm.edu.my` — `studentid`, `uuid` all present, no `force_logout`. Data is clean → ruled out the data-based theories (missing uuid, malformed studentid, stuck force_logout).
- Confirmed Firestore rules block unauthenticated reads, so per-student checks have to happen in the Firebase console.
- Traced every redirect-to-`/login` path across `router/index.js`, `store/auth.modules.js`, `main.js`, `LoginPage.vue`.

**Two root causes found**

| # | Cause | Fix | File |
|---|-------|-----|------|
| 1 | `signInWithRedirect` (added 2026-06-16) never completes on iOS Safari / in-app WebView — cross-origin bounce to `*.firebaseapp.com` authDomain wipes the pending-redirect state, `getRedirectResult()` returns null → back to login | Reverted to **popup-primary** for everyone; redirect kept only as popup-blocked fallback | `LoginPage.vue` |
| 2 | Device-independent infinite `/` ↔ `/login` bounce: Vuex seeds `loggedIn` from localStorage, but the guard's `!current` branch cleared only localStorage, not the store → `LoginPage.created()` reads stale `loggedIn=true` and pushes back | Guard now `store.dispatch('auth/logout')` in both the `!current` and `catch` branches → clean fail to login | `router/index.js` |

💡 The "authDomain never changed, yet the bug is new" puzzle resolved: authDomain was *always* cross-origin, but popup never depended on redirect-state surviving. The 2026-06-16 popup→redirect switch is what made the always-cross-origin authDomain suddenly fatal under ITP/WebView.

⚠️ Two different "tokens" in this system — don't conflate: **Firebase Auth session** (browser IndexedDB, drives the login loop) vs **`token_mystudent`** (Firestore, Cloud-Function-minted, only for borang perubatan upload, already decoupled from login).

🧩 Investigated a suspected #3 (encrypted-vs-plaintext `studentid` in the store after fresh login) — verified against the Firebase SDK source that `docSnap.data()` returns a fresh object per call, so the store already keeps the *plaintext* studentid. NOT a bug, no fix needed.

📝 Both fixes are in the working tree on `development`, uncommitted. Memory updated in `-Applications-Sites-mystudentvue` store (popup memory corrected, new session-desync memory added).

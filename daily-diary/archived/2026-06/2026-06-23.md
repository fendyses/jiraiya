## 2026-06-23 (Afternoon - 12:34) - Windmill Logo Swap & Two 3D Overlay Bugs

### Session Summary
- Type: Bug Fix / UI Polish
- Focus: JIRAIYA dashboard 3D village — swap the windmill CR emblem, fix a nametag that detached during the cart animation, and fix the CR dialogue box landing in the wrong corner.
- Outcome: All three shipped. Logo replaced, name now rides the character onto the wheelbarrow, and the CR bubble pins onto the logo correctly on Retina.

### What Was Fixed

1. **Windmill CR emblem swapped to Uchiha logo** — The old `sharingan.png` was deleted from the repo, leaving the windmill signboard texture dangling. Repointed `village3d.js` to load `assets/pics/uci.png` instead (the new Uchiha emblem). One-line texture path change.

2. **Cart-visit nametag flew "elsewhere"** — Root cause was an architecture seam: the visible characters are **3D models** (three.js), while their nametag/HP bar are **2D Phaser** UI pinned to an invisible logic-sprite. Normally the 3D model follows the sprite so they line up. But the cart-visit director (`updateCartVisit`) detaches the model — it walks to the wheelbarrow and hops on top on its own — while the sprite keeps wandering. The nametag stayed glued to the stray sprite. **Fix:** added `world3DToPhaser()` (inverse of the existing `phaserTo3D()` raycast), published the visiting model's projected feet position to `window._CART_VISIT` each frame, and had `syncUI()` pin the name/HP/bubble to that while the visit is active. Name now rides the character up onto the cart and back down.

3. **CR dialogue box appeared top-left instead of on the logo** — Coordinate-space mismatch. `_bg3dProject` returned pixels in the *logical* canvas size (`W`/`H`), but `showCRBubble` then re-divided by `canvas.width`/`canvas.height`, which is `W × devicePixelRatio` on a Retina display — roughly halving the position and dragging the bubble up-left. **Fix:** `_bg3dProject` now returns normalized 0–1 coords and `panels.js` multiplies by the displayed CSS size — fully resolution-independent.

4. **Logo centering nudge** — Moved the CR signboard right from `x=-8.5` to `x=-8.3` to sit centered on the windmill tower, and updated the matching `showCRBubble` anchor to `-8.3` so the dialogue stays pinned to the logo. The click hitbox follows automatically (raycasts against the moved mesh).

### Key Insights & Learning
- **The recurring theme today was the 2D/3D seam.** Both bugs (#2 and #3) came from the same root: a 2D overlay assuming it shares the 3D model's coordinate space when it doesn't. The cart bug was a *position* decoupling; the bubble bug was a *resolution* decoupling. Both fixed by being explicit about projecting 3D→screen and which pixel space we're in.
- **devicePixelRatio bites silently.** The CR bubble math looked correct on paper and probably worked on a 1× display — only the Retina 2× buffer exposed the halving. Normalized coords are the safe default for any 3D→DOM overlay.

### Notable Moments
- Diagnosed the cart nametag bug purely from reading the render loop — traced how `updateCartVisit` overrides `m.position` independently of the Phaser sprite, which made the detachment obvious.

### Looking Forward
- Reload-and-verify on the live dashboard (Retina) to confirm the bubble lands dead-on the logo and the logo centering feels right — both are single-number tweaks if not.

## 2026-06-23 (Afternoon - 13:50) - MyStudent: Medical-Form Token & Login-Loop Fixes

### Session Summary
- Type: Bug Fix / Debug / Decision
- Project: `/Applications/Sites/mystudentvue` (branch `development`)
- Focus: Trace and fix the "SSO invalid" medical-form error and the widespread "after login → back to /login" loop.
- Outcome: 3 commits built + deployed to Firebase Hosting (`mystudentuitm.web.app`). Root causes found across web app **and** a Cloud Function.

### Main Topics Discussed

1. **Stale token cache → "SSO invalid"** (commit `bd3c586`)
   - `src/firebase.js` enables `enableIndexedDbPersistence`; `getDoc` on `token_mystudent/{id}` kept serving the **stale cached** token after renewal (modal showed `22 April 2026`). Logout clears `localStorage` but not Firestore's IndexedDB cache → loop.
   - Fix: read the token via `getDocFromServer` (fallback to `getDoc` only when offline).

2. **App-wide login gate → login loop** (commit `7e8a31f`)
   - `main.js` startup check force-logged-out on `!hasToken || force_logout`. The token is only for borang-C upload but it gated **all** app login. `force_logout` was set `true` by AdminPage and **never reset** → permanent loop.
   - Fix: removed the `!hasToken` gate; kept a **one-time** `force_logout` that resets the flag before logout.

3. **Where the token is minted (Cloud Function dig)**
   - Used `gcloud`/`gsutil` to pull deployed source: Python functions `mystd_login_new` (on `mystudentlog/{id}` create) / `mystd_login_update` (on update). Mint a 14-day JWT into `token_mystudent` **only if** `pelajar/{id}` exists. They write field **`exp`** (unix seconds), not `expiresAt`.
   - Verified via Firestore REST: student `2026239202` has a profile but **no `mystudentlog` and no token** → function never fired for them.
   - Bugs to hand the colleague: `exp` vs `expiresAt` mismatch, `mystd_login_new` try/except commented out, hard-coded JWT secret.

4. **Scoped in-page token refresh** (commit `54b77de`)
   - Reverted the "write `mystudentlog` on every login" idea (too many function invocations). Scoped the refresh to `MedicalForm.vue`: detect expired/missing token (`getTokenExpiry` reads `exp`), show modal with **"Muat Semula Token"** button → writes `mystudentlog/{id}` to trigger the function, polls `token_mystudent` (~25s) until the fresh token lands.

### Key Insights & Learning
- 💡 A **feature-scoped credential** (medical-form token) was wrongly used as an **app-wide login gate** — that single conflation caused most of the loop complaints.
- ⚠️ Firestore offline persistence (`getDoc`) can silently serve stale docs; use `getDocFromServer` for anything that must be fresh, and remember logout does NOT clear the IndexedDB cache.
- 💡 The `exp`/`expiresAt` mismatch was neutralised on the client (read `exp`), so the backend rename is no longer required.

- Optional: also check `exp` at **send time** and route a 401 to the same refresh modal (currently only checks on page load).
- Hand the Cloud Function fixes (error handling, secret) to the colleague's separate functions repo.

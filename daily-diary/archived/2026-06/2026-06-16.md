## Session: Dashboard — Drag-and-Drop for Characters & Pets

Worked in `agents/dashboard.html` (JIRAIYA Live Agent Ecosystem, Phaser 3 scene).

---

### 1. Mouse Drag for Agent NPCs

Fendy asked if characters could be picked up and placed elsewhere on screen with the mouse.

- `AgentNPC` sprites made interactive (`setInteractive()` + `scene.input.setDraggable()`)
- Added `dragging` flag — `pointerover`/`pointerout` swap cursor to `grab`/`default`
- `update()` short-circuits to just `syncUI()` while dragging, so AI movement/state logic pauses
- Scene-level `dragstart` / `drag` / `dragend` handlers on `GameScene`:
  - `dragstart`: freezes velocity, forces `idle` state, sets `summoned=true` (reuses the existing flag that suppresses autonomous movement during the summon event), bumps sprite depth to 9999, cursor → `grabbing`
  - `drag`: clamps new position to the walkable bounds (same bounds as normal wandering)
  - `dragend`: restores depth by Y-position, cursor → `default`, clears `summoned`, calls `pickTarget()` to resume normal AI from the drop point

---

### 2. Extended to Pets

Same treatment applied to `PetNPC` (fox/penguin/cat):

- Constructor: `setInteractive()`, `setDraggable()`, cursor events
- `update()`: early return if `dragging`
- Drag handlers generalized to check `this.npcs` first, then fall back to `this.pets` — pets update `baseX`/`baseY` (their movement anchor) rather than sprite position directly, since `update()` applies a bob offset on top of `baseX/baseY`
- `dragend` calls `pet.pickTarget()` to resume wandering from the new spot

---

### Notes
- Single set of `dragstart`/`drag`/`dragend` listeners on the scene handles both agents and pets — avoided duplicating handler logic per entity type.

📌 Focus  : Two student-reported login issues in `/Applications/Sites/mystudentvue`
✅ Result : Diagnosed and fixed both; code changes applied, not yet deployed/tested live

**Issue 1 — `auth/popup-closed-by-user` (student 2025127885@student.uitm.edu.my)**
- Cause: `signInWithPopup` doesn't survive mobile browsers / in-app WebViews (links opened from WhatsApp, email, etc.) — popup opens and dies instantly. Not an account-activation problem.
- Fix in `src/views/LoginPage.vue`: detect mobile/WebView via user-agent → use `signInWithRedirect` instead of popup. Desktop keeps popup but falls back to redirect on `popup-closed-by-user`/`popup-blocked`. Added `processGoogleRedirectResult()` in `mounted()` to consume the redirect result.

**Issue 2 — Login loop after clicking "sini" in sponsor/penaja popup (student 2026602718@student.uitm.edu.my)**
- Two compounding bugs:
  1. `src/services/auth.service.js` `login()`: on first-ever login (no `uuid` yet on the student's Firestore `pelajar` doc), a new uuid is generated and written to Firestore but never assigned to the `data` object → `JSON.stringify` drops the `undefined` → localStorage user has no `uuid`.
  2. `src/views/HomePage.vue`: the "sini" link in the `DialogSponsor` modal was a plain `<a href="/sponsor">`, causing a full page reload. On reload, `src/store/auth.modules.js` runs `Crypto.AES.decrypt(studentid, uuid)` at module load — missing `uuid` → empty string → fails `length < 10` check → force logout (`localStorage.clear()`) → redirect to `/login`. Looked like a permanent loop because every device hits the same "first login" path.
- Fix: added `data.uuid = uuid;` in the uuid-generation branch of `auth.service.js`; changed the modal link to `<router-link to="/sponsor">` to avoid the unnecessary hard reload.

| File | Change |

| `src/services/auth.service.js` | persist generated `uuid` into `data` before localStorage save |

💡 Any future "kicked back to login right after one specific click" report → check for `<a href>` vs `<router-link>` first, then check the student's Firestore `uuid` field.

**nilam — Renewal application "looks like it doesn't complete" (`/assigns/7709`)**

📌 Focus  : User complaint about a renewal application flow "looking weird/incomplete" in `/Applications/Sites/nilam`
✅ Result : Not a backend bug — confirmed via DB query (read-only `nilams` prod DB) that the renewal flow works as designed, but found a real UX gap and fixed it

**Root cause of the confusion**
- App #7709's original agreement reached `executed-document` on 2026-06-15 12:07, then 32 min later flipped back to `pending-approval-by-la` — that's `MyApplicationController::storeExtension` (line 532) doing its job: snapshot the executed agreement into `application_renews`, then reset the live row to restart the full approval pipeline (vetting → MEU → LPU → signing) from scratch.
- The actual UX bug: `vettingDocuments()` relation (`app/Application.php:263`) is a plain `hasMany` keyed on `application_id` with no concept of "cycle." So the `/assigns/{id}` show page kept displaying the **previous cycle's** vetter/PIC comment history mixed in with the (empty) new cycle — looked like vetting already happened while the system was simultaneously asking to "Assign Vetter/Drafter" again.
- Secondary gap: `ApplicationStatusLog` rows created by `storeExtension`/`addendumExtension` had no `note`, so the "Application Status Log" tab (`resources/views/applications/show.blade.php:350`) showed a renewal-reset entry that looked identical to a brand-new application's first submission.

**Fixes applied (not yet committed)**
| File | Change |
|---|---|
| `app/Http/Controllers/AssignController.php` (`show()`) | Computes `$cycleStartedAt` (latest `pending-approval-by-la` status log), splits `vettingDocuments` into `currentCycleVettingDocuments` / `previousCycleVettingDocuments` |
| `resources/views/assigns/show.blade.php` | "History from previous cycle (N)" collapsible button now rendered *above* the current-cycle table (per Fendy's request), `mt-2` padding added between them |
| `app/Http/Controllers/MyApplicationController.php` (lines ~551, ~672) | `storeExtension`/`addendumExtension` now write a `note` ("Application submitted for Extension/Renewal" / "...for Addendum") on the status log row |

⚠️ DB is read-only (prod `nilams` via `antartika.uitm.edu.my`) — confirmed no UPDATE/backfill possible. Existing log rows (incl. #7709's) won't retroactively get a note; fix only applies to renewals submitted going forward.

💡 Pattern to remember: any `hasMany` relation on `Application` that isn't scoped by cycle (vetting docs, possibly others) will silently carry history across renewal resets. Worth checking `documents()`/other relations if similar complaints come in.

- In jiraiya: added `.aider.conf.yml` (auto-loads `CLAUDE.md`, `master-memory.md`, `main/main-memory.md` as read-only context every session — closest equivalent to Claude Code's CLAUDE.md auto-read) and `.aiderignore`
- Hit a real performance bug during testing: first live query took ~2.5 min. Root-caused two separate issues:
  1. qwen3.5 is a hybrid-thinking model — Ollama's `/api/chat` defaults to extended `thinking` mode, burning tokens on internal monologue even for trivial prompts. Fixed via `--reasoning-effort none` (litellm maps this to Ollama's `think:false` param) — confirmed via `curl ... -d '{"think":false}'` direct to the Ollama API first, then traced the equivalent litellm/aider flag from source (`litellm/llms/ollama/chat/transformation.py`)
  2. aider's repo-map choked on jiraiya's 14,353 git-tracked files — turned out 13,939 of those are dashboard sprite PNGs under `agents/assets/`. Excluded via `.aiderignore` (`agents/assets/`, `data.ms/`, `*.zip`)
- Baked both fixes into defaults (`.aider.conf.yml` for jiraiya, the `qwen` wrapper for every other repo) so Fendy doesn't need to remember the flags
- Explained the two-layer Ollama lifecycle Fendy asked about afterward: the model itself auto-unloads from RAM/GPU after ~5 min idle (`OLLAMA_KEEP_ALIVE` controls this), but the background Ollama service stays running and ready — it doesn't hibernate, only the loaded model does

💡 Pattern to remember: for any local Ollama reasoning/hybrid-thinking model wired into a tool via litellm, check `reasoning_effort`/`think` support first — it's usually the single biggest latency lever, bigger than model size or repo-map scope.

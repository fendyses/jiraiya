## 2026-06-29 (Morning - 09:33) - MyStudent Bug Investigations & Fixes

### Session Summary
- Type: Bug Investigation + Fix
- Focus: Three separate student-reported issues on MyStudent (mystudentvue)
- Outcome: All three issues diagnosed and fixed

### Main Topics Discussed

1. **Student 2026494406 — /sponsor page dark/inactive**
   - Student navigates to `/sponsor` via a `<router-link>` inside Bootstrap's `DialogSponsor` modal
   - Root cause: Bootstrap modal backdrop (`<div class="modal-backdrop">`) + `body.modal-open` class stay in DOM when Vue Router navigates away — no page reload, no cleanup
   - `/sponsor` renders fine but sits behind the dark backdrop overlay
   - Fix: added `router.afterEach()` in `router/index.js` to remove `.modal-backdrop` and clear `modal-open` styles on every navigation
   - This covers all future modals with in-page links — not just `DialogSponsor`

2. **Student 2026103875 — Medical form image upload fails**
   - Error: "Gagal memuat naik gambar!" → "Gagal menghubungi pelayan. Token mungkin telah tamat tempoh"
   - Initial assumption: expired token → checked Firestore `token_mystudent` → token exp = 1783584317 (valid until July 9, 2026)
   - Real cause: Chrome's **Private Network Access (PNA)** policy — `api.uitm.edu.my` resolves to a private/internal UiTM IP. Chrome blocks public-site → private-network requests without "Local network" permission
   - Browser reports this block as `TypeError: Failed to fetch` — same as a network error, no HTTP status code visible
   - Fixes applied to `MedicalForm.vue`:
     - Pre-upload server token re-read (catches genuinely expired tokens before hitting the API)
     - 401 errors now show the token refresh modal instead of an alert
     - "Failed to fetch" now shows: *"Sila klik ikon 🔒 → aktifkan Local network → muat semula"* instead of the misleading "token tamat tempoh" message
   - Student fix: enable "Local network" permission in browser lock icon menu

3. **Students 2026231952, 2026228388, 2026267448 — Cannot login (Google/SSO)**
   - Error shown: **"Invalid SSO state."**
   - Not a Google login bug — triggered by `processMydigitalidSSOCallback()` on `mounted()`
   - Root cause: students open MyStudent link from WhatsApp or similar in-app browser → click "Login with MyDigital ID" → `kc_login_state` stored in in-app WebView's localStorage → Keycloak redirects back in Safari/Chrome (different localStorage context) → state mismatch
   - Secondary bug: on state mismatch, the stale `?code=...&state=...` URL was never cleaned → every refresh/retry re-triggers the error → students permanently stuck
   - Other causes that produce the same error: browser back button after SSO, multiple tabs, logout-then-relogin from history URL, page refresh mid-redirect
   - Fixes applied to `LoginPage.vue`:
     - On state mismatch or missing verifier: `window.history.replaceState()` cleans the URL + removes stale localStorage keys immediately
     - Error message changed to Malay: *"Sesi log masuk SSO tidak sah atau telah tamat. Sila cuba log masuk semula."*
     - Added `detectInAppBrowser()` — detects WhatsApp, Instagram, Facebook, Twitter, Line, WeChat, TikTok, iOS WebView, Android WebView
     - Added warning banner above login buttons when in-app browser detected
     - `externalBrowserName` computed — platform-aware: iOS → Safari atau Firefox, Huawei → Browser Huawei, Chrome atau Firefox, Android → Chrome atau Firefox
     - "🌐 Buka dalam Browser" button calls `window.open(url, '_blank')` to attempt external browser launch

### Key Insights & Learning
- Bootstrap 5 modal backdrop is appended to `document.body` and must be manually cleaned on SPA navigation — Vue Router doesn't know about it
- Chrome's PNA policy is a silent killer — blocks private-IP requests with "Failed to fetch" and no HTTP status, making it look like a token/network error
- SSO state in localStorage doesn't survive cross-browser-context redirects (in-app → real browser). Cleaning the stale URL on failure is as important as the fix itself
- Multiple different root causes (in-app browser, multiple tabs, back button, history URL) all produce the same "Invalid SSO state" — the URL cleanup fix covers all of them

- Tell student 2026103875 to enable "Local network" in browser settings
- Tell students 2026231952 / 2026228388 / 2026267448 to open directly in Chrome/Safari (not from WhatsApp) — after deploy, refreshing will auto-clear the stuck state

## 2026-06-29 (Morning - 11:22) - Homepage Banner Not Appearing

### Session Summary
- Type: Bug Investigation
- Focus: Banner carousel not rendering on homepage
- Outcome: Root cause narrowed down — awaiting console log from user to confirm

### Main Topics Discussed

1. **Homepage banner carousel not appearing**
   - Template condition was `v-if="settings.homepagevideo == ''"` — only passed for exact empty string
   - Bug: if `homepagevideo` is missing from Firestore or is `null`/`undefined`, condition evaluates false → carousel hidden
   - Fix applied: changed to `v-if="!settings.homepagevideo"` — covers empty string, null, and undefined
   - Added `console.log('[HomePage] settings.homepagevideo | settings.banner')` in mounted() to debug
   - Firebase confirmed `banner` array IS populated with valid items (eprofiling.png, kalendar2026.jpeg etc.)
   - Banner items have empty `title` and `description` but valid `url` — image should still render regardless
   - **Remaining uncertainty**: `homepagevideo` field value in Firestore unknown at runtime — if non-empty, it overrides the carousel
   - **Next step**: user to check DevTools console for `[HomePage]` log line

### Key Insights & Learning
- `== ''` vs `!value` — Firestore returns `undefined` for missing fields, not `''`. Always use falsy check for "empty or absent" guards
- Bootstrap carousel items with empty `title`/`description` still render fine — `<img :src="data.url">` is unconditional
- When banner doesn't show: two possible causes — (1) `homepagevideo` truthy in Firestore, (2) `banner` empty/undefined

- If `homepagevideo` has a value → need to clear it in Firestore or admin panel

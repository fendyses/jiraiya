## 2026-07-10 (Morning - 11:57 AM GMT+8) - Dashboard: Diary Book Font + Fixed Real Layout Bug in Date List

### Session Summary
- Type: Work (Bug Fix + Polish)
- Focus: Dashboard diary book (`agents/dashboard.php`) readability — font clarity and a persistent "cut off" rendering issue in the date list
- Outcome: Done — root cause found and fixed via a headless-browser repro (not guesswork); font swapped to a legible serif; same latent bug preemptively fixed in the CR Records book

### Main Topics Discussed
1. Fendy flagged the diary book's date list (`06-07-2026` etc.) as hard to read. Root cause: `.book-date-item` used `IM Fell English`, a historical typeface with old-style figures (digits with varying heights, like lowercase letters) — genuinely bad for scanning dates. Swapped it and the surrounding diary typography (`.book-title`, `#bookContent`, headings) from `IM Fell English`/`IM Fell English SC` to `Lora` across `agents/css/dashboard.css`, added the `Lora` Google Fonts import to `agents/dashboard.php`.
2. First fix attempt: added a bottom `mask-image` fade to `#bookDateList`/`#bookContent` to soften the hard `overflow-y:auto` clip. Fendy reported it "still the same" after a hard refresh — this ruled out browser caching as the cause, which had been my working theory (dashboard.css/panels.js had no cache-busting query string, so also fixed that by adding `?v=<?= filemtime(...) ?>` to both in `dashboard.php`, worth keeping regardless).
3. Second fix attempt: added `snapListHeight()` in `agents/js/panels.js` to cap the list's `max-height` to a whole number of rows, reasoning a partial row was being sliced mid-line. Still reported as unresolved, with a key clue from Fendy: *"it doesn't happen like this before when the list is fewer"* — pointed straight at a layout bug that only manifests once content overflows.
4. Stopped guessing from screenshots and built an isolated reproduction instead: found Playwright bundled inside a globally-installed `@executeautomation/playwright-mcp-server` package (`node_modules/playwright-core` + `playwright`), used it headlessly to load a standalone copy of the real diary-book markup + the actual `dashboard.css`/`panels.js` logic, and inspected computed styles directly.
   - Diagnosis revealed the real bug: `#bookDateList` is `display:flex;flex-direction:column`, and `.book-date-item` never had `flex-shrink:0`. With few entries, total content height stayed under the box's height and flex-shrink never triggered. Once the diary grew past ~20 dates, flex-shrink started **compressing every row** to cram them all in — rows measured ~13.7px tall instead of their natural ~27-30px, giving the cramped/hazy "cut" look Fendy kept seeing, worst at the last row.
   - Confirmed the fix (`flex-shrink:0` on `.book-date-item`) via a second headless screenshot: rows render at natural height (~30px), the list genuinely overflows (`scrollHeight` 919px vs `clientHeight` 501px) and scrolls properly, `snapListHeight()` now computes correct row counts since it's working with real measurements instead of the compressed ones.
5. Follow-up request: apply the same font legibility fix to the CR Records book. While in the CSS, noticed `#crFileList`/`.cr-file-item` had the exact same missing-`flex-shrink:0` setup as the diary list — fixed it preemptively before it could bite the same way as the CR log grows. Swapped `.cr-book-title`, `#crBookContent`, `.cr-date-header`, and the inline-styled archive header label in `dashboard.php` from `IM Fell English`/`SC` to `Lora` too.

### Key Insights & Learning
- A user-reported "it still looks the same" after two genuine CSS fixes was the signal to stop iterating blind on screenshots and reproduce the bug directly — a throwaway Playwright harness (loading the real CSS/JS files via `file://`) took a few minutes and immediately surfaced the actual cause (flex-shrink compression) that no amount of CSS reasoning from the screenshot alone was going to find.
- `display:flex;flex-direction:column` + `overflow-y:auto` on a scrollable list is a common trap: without `flex-shrink:0` on the list items, the list will silently switch from "scroll when it overflows" to "compress everything to fit" once content grows past the container — and it works perfectly fine (misleadingly) while the list is still short.

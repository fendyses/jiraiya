**Sakura CLI — Added Groq as a Second Provider (not xAI Grok)**

📌 Focus  : User asked to add "Grok api" with a key `gsk_...`, wanted it usable in `sakura`/`ai`
✅ Result : Discovered the key was actually a **Groq** key (not xAI's Grok); refactored the CLI to support multiple providers and added 4 real Groq models

### What Was Done

1. **Caught a naming mix-up before wiring anything wrong**
   - Key prefix `gsk_` = Groq (groq.com, LPU inference hosting) format; xAI keys start with `xai-`
   - Verified by calling `api.groq.com/openai/v1/models` with the key — it authenticated (200) and returned Groq's real hosted models (Llama, Qwen, GPT-OSS, Whisper) — no "grok" model anywhere, confirming it's unrelated to xAI
   - First (wrong) attempt added `x-ai/grok-code-fast-1` via OpenRouter using the *existing* OpenRouter key — reverted once the Groq/Grok confusion was confirmed with the user

2. **Refactored `models.json` to support multiple providers**
   - Added top-level `"providers"` map: `openrouter` (existing) + `groq` (new), each with `base_url` + `api_key_env`
   - Every model entry now carries a `"provider"` field
   - Added 4 real Groq-hosted models:

   | Alias | Model ID | Best for |
   |-------|----------|----------|
   | `llama70` | llama-3.3-70b-versatile | General chat, reasoning, fast inference |
   | `llama8` | llama-3.1-8b-instant | Fast responses, low latency |
   | `qwen32` | qwen/qwen3-32b | Reasoning, code gen, tool use |
   | `groqoss` | openai/gpt-oss-120b (Groq-hosted) | General chat, reasoning, tool use |

3. **Updated `chat.js`, `sakura.js`, `test-models.js`** to resolve API key + base URL per-model from its `provider`, instead of one hardcoded global OpenRouter key. `chat.js`'s `--list` now shows `(provider)` next to each model.

4. **Added `GROQ_API_KEY` to `.env`**, verified all 4 Groq models respond 200 OK live through the real CLI code path (`node chat.js llama70 "..."` etc.)

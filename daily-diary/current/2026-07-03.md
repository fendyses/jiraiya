**Sakura CLI — Interactive Model Picker Built on the Wrong File, Then Fixed on the Real One**

📌 Focus  : User wanted `sakura`'s `models` command to become an arrow-key toggle picker (with Cancel) instead of a flat list, and asked why typing `models` "just lists but I can't choose"
✅ Result : Discovered two divergent copies of Sakura existed; the actually-running one lives at `/Users/pairofspades/Documents/Ai/sakura/sakura.js` (a Grok-focused CLI — grok-4.20/4.3/multi-agent/build-0.1 + live free-model fetch from OpenRouter), not the copy at `jiraiya/sakura/` that had been edited first. Rebuilt the picker on the correct file, found and fixed a real bug along the way.

### What Was Done

1. **First pass (wrong target)** — added a numbered model list + `model <n>` switch command, then a full arrow-key picker with a Cancel entry, to `/Applications/Sites/jiraiya/sakura/sakura.js`. Also added a "Grok 4.3" entry to that file's `models.json`. User reported the live `sakura` command didn't behave this way at all.
2. **Root-caused the mismatch** — `which -a sakura` → `/Users/pairofspades/.local/bin/sakura`, which turned out to be a shim (`require('/Users/pairofspades/Documents/Ai/sakura/sakura.js')`) pointing at a completely different, simpler implementation. That real file has no `read_file`/`write_file`/`run_command` tool-calling loop at all — only manual `load <path>` / `@file` context injection. **Corrected an earlier wrong answer given this session**: Sakura is *not* currently agentic in the tool-calling sense; that description matched the unused copy, not what actually runs.
3. **Rebuilt the picker on the real file** — added `modelItems()` (merges the 4 static Grok models with the dynamically-fetched OpenRouter free-model list into one numbered set) and `interactiveModelPicker()` (raw-mode keypress capture: ↑/↓ move, Enter select, Esc cancel, plus a selectable Cancel row). Kept `model <id>` and added `model <n>` for direct numeric switching.
4. **Found and fixed a real bug during testing**: closing the active `readline.Interface` to hand the terminal to raw-mode keypress capture also fired that interface's `'close'` handler — which was written to always call `process.exit(0)` (meant for genuine Ctrl+D/session-end). Typing `models` was silently killing the whole session. Fixed with a `switchingModel` flag so the close handler only exits on a real user-initiated close.
5. **Verified for real, not just by reading code** — wrote a throwaway Python harness (`pty.openpty()` + `subprocess.Popen`) to run `sakura.js` under an actual pseudo-terminal and inject real arrow-key escape sequences (`\x1b[B`, `\r`, `\x1b`), since a piped/non-TTY stdin can't exercise raw-mode input at all. Confirmed: cursor moves correctly, selection switches the model, re-opening the picker remembers the current selection, Esc cancels cleanly without changing state, and `model 2` direct-switches to Grok 4.3.

### 💡 Key Notes
- Multiple Node CLI copies with the same name (`sakura.js` in three different folders across two sessions) is an active risk — always confirm `which -a <cmd>` → shim target → actual file before editing, not just grep for the filename.
- A `readline.Interface`'s `'close'` event fires on *any* `.close()` call, not just user-initiated ones — any code that intentionally closes/reopens the interface (e.g. to toggle raw mode for a custom picker) needs a guard flag, or it'll be indistinguishable from Ctrl+D.
- Piped stdin can't validate raw-mode/arrow-key behavior — a real pty (Python's `pty` module, or `script`) is required to actually test TTY-dependent interactive UI in a CLI.

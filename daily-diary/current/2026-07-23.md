## 2026-07-23 - NILAM: "Pending LPU Approval" status bug investigation

### Session Summary
- Type: Debugging / root-cause analysis (NILAM repo)
- Focus: MEU Secretariat complaint — Approved MEU page shows "Pending LPU Approval" for every application, including LPU-notification-only ones.
- Outcome: Root cause found and confirmed against live DB. Fix proposed but NOT yet applied.

### The 3 statuses that matter
| ID | slug | Shows on screen | Meaning |
|----|------|-----------------|---------|
| 12 | `submit-for-lpu-notified` | Submitted for LPU **Notification** | LPU just needs to be told |
| 13 | `pending-for-lpu-approval` | Pending LPU **Approval** | LPU must approve it |

MEU decides which via the `lpu_approval` flag (`0` = notify only, `1` = needs approval).

### What should happen vs. what happens now
| App's `lpu_approval` | Correct status | Status it gets now | OK? |
|----|----|----|----|
| `0` (notify only) | 12 (Notification) | **13 (Approval)** | WRONG |
| `1` (needs approval) | 13 (Approval) | 13 (Approval) | ok |

Everything gets stamped 13 → all rows read "Pending LPU Approval".

### Root cause (regression from NILAM commit `427649b` "bug - renewal mou", 5 Jun 2026)
Two small mistakes combined:
1. `LpuApprovalController::meeting()` has always **logged** status 13 for *every* app in the batch (even notify-only). Originally harmless because the app's real status field was left at 12 for notify apps.
2. That commit added a global model hook in `app/ApplicationStatusLog.php` (~line 53): *"whenever a status-log row is created, force the app's status to match the log."*

Combined: the log said 13 for notify apps → the hook forced the real status to 13 → correct 12 overwritten. The same commit also flattened `meeting()` to always set 13, cementing it.

One line: **a new "keep status in sync with the log" rule trusted a log that was already wrong.**

### Why only MEU Secretariat noticed
LPU pages display `lpu_approval == 1 ? status->name : 'LPU Notified'`, hiding the wrong status. Only the Approved MEU page (`meuapprovals/approved.blade.php`) shows the raw status name.

### Damage
| | Count |
|---|---|
| Notify-only apps wrongly showing "Pending LPU Approval" | **96** |
| MEU meetings affected | 5 (160=38, 158=37, 159=15, 156=4, 161=2) |
| All under one LPU batch | LPU meeting #29 |

Target set for correction: `lpu_approval = 0 AND application_status_id = 13`.

### Key Insights & Learning
- The status-log "sync" hook is globally powerful — any caller that logs a placeholder status will now have that placeholder become the real status.
- The bug was masked everywhere except the one page that renders the raw status name.

## 2026-06-17 (Night - 12:25 AM) - Fixing and Speeding Up Local Qwen/Aider Setup

### Session Summary
- Type: Tooling / Troubleshooting
- Focus: Fix broken `qwen` (aider + Ollama) CLI after swapping qwen3.5 → qwen2.5, then optimize for speed
- Outcome: Fully fixed, reconfigured, and benchmarked — `qwen` now runs on a leaner, faster local model setup

### Main Topics Discussed
1. **Bug**: Fendy deleted qwen3.5 and installed qwen2.5-coder:7b, but `qwen` was still erroring — both `.aider.conf.yml` and the `~/.zshrc` `qwen()` wrapper had `ollama_chat/qwen3.5:latest` hardcoded. Updated both to the installed model and verified a live aider session loaded correctly.
2. **Speed optimization**: Fendy asked how to make it run/think faster. Diagnosed hardware (Apple M2, 16GB RAM) and identified four concrete levers: trim context length, use a smaller model, disable aider's repo-map, raise `OLLAMA_KEEP_ALIVE`.
3. **Implementation**: Pulled `qwen2.5-coder:3b`, deleted `qwen2.5-coder:7b` per Fendy's request, built a custom `qwen-aider` Ollama model (3b base + `num_ctx 8192` instead of default 32768), set `map-tokens: 0` in `.aider.conf.yml` and `--map-tokens 0` in the wrapper, added `OLLAMA_KEEP_ALIVE="30m"` to `~/.zshrc`. Removed every hardcoded model reference in favor of the single `qwen-aider` name.

### Key Insights & Learning
- Two separate hardcoded model references existed (project `.aider.conf.yml` AND the global `~/.zshrc` wrapper) — both needed updating, not just one.
- Ollama lets you layer custom parameters (like `num_ctx`) onto an existing model via a Modelfile + `ollama create`, producing a new named model without re-downloading weights.
- Warm inference time dropped to ~19s for a trivial prompt after applying all four optimizations (vs. noticeably slower with the 7b model and full repo-map/context).

### Looking Forward
- Could compare 3b vs 7b code-quality tradeoff on a real coding task if `qwen-aider` proves too weak for harder edits.
- No open follow-ups — setup verified working end-to-end.

## 2026-07-21 (Morning - 09:21 to 10:15) - MyAlumniCard: Why 15,819 Alumni Records Lost `tarikh_create`

> ⚠️ Entry reconstructed on 2026-07-22 from the session transcript. The original session ended without a diary save, so this was invisible to `/recall` for a day. Timings are from the transcript; findings are quoted as delivered.

### Session Summary
- Type: Investigation / incident analysis (MyAlumniCard — `/Applications/Sites/myalumni-angular`)
- Focus: Find out why `tarikh_create` is missing for a large share of the Firebase `alumni` collection
- Outcome: Root cause established with a control case; full-collection scan quantified the damage; 3-page PDF report produced. **No code fix applied — still broken in production.**

### Main Topics Discussed
1. Fendy reported that many registered users have no `tarikh_create` in the Firebase `alumni` collection, giving alumni id `1353151` as an example, and asked when and why this started.
2. Investigation ran read-only against Firestore via authenticated `gcloud` CLI — full scan of all 286,207 documents rather than a sample.
3. Scope quantified: **15,819 records created since 17 April 2026 are missing the field — a 100.0% failure rate.** Of those, 14,466 are completed registrations (`register = 1`); the remaining 1,354 were created but never completed.
4. Month breakdown: Apr 2026 (from 17th) 6,156 of 6,157 · May 6,001 of 6,001 · Jun 3,153 of 3,153 · Jul (to 21st) 509 of 509. Records created *before* the cutoff show only a 6.1% miss rate — a hard break, not a worsening trend.
5. Root cause traced to the hardcoded JWT bearer token in `register.component.ts:175`. The old token expired Mon 20 Apr 2026 11:27 MYT; registration started failing and it was replaced the same afternoon in commit `aa01c8b` *"update token -register"*, deployed 14:54 MYT — roughly a 3.5-hour outage, handled reasonably.
6. Decoding both tokens showed the replacement belongs to a **different application**: `user`/`app` went from `alumnai` to `myatp`, and `scope` from `/alumnai` to `/alumnai/semak`. Upstream still authorises the call, returns a valid `alumni_id`, and registration completes — but UiTM's system only stamps `tarikh_create` for requests arriving as the `alumnai` application.
7. Control case that confirms it: the **attendance** flow uses a separate token, renewed later on 14 May 2026, which kept the `alumnai` identity (`attendance.component.ts:102`). That path was never affected. Only the register flow changed identity, and only the register flow lost the field.
8. Timeline: Fri 17 Apr 21:59 MYT — last `tarikh_create` ever written · weekend, no traffic · Mon 20 Apr 11:27 — token expires · Mon 20 Apr 14:54 — `myatp` token deployed · onward — registrations succeed, stamping never resumes.
9. Fendy asked for the finding as prose rather than a report page, then as a PDF. Delivered to `~/Desktop/tarikh_create-incident-report.pdf` (3 pages, A4, 234 KB).

### Key Insights & Learning
- **The outage fix swapped a visible failure for an invisible one.** Restoring service with whatever valid token was on hand was a sound instinct under time pressure; the missed step was checking that the replacement carried the *same app identity*, not merely that it authorised.
- **It stayed hidden for three months because the app neither writes nor reads the field.** Registration records only `formData.register = 1` (`profile.service.ts:218`) — a flag with no date. `tarikh_create` is written entirely by UiTM upstream, so when upstream stopped there was no error, no failed write, and no local signal of any kind.
- **Nothing breaks functionally** — `tarikh_create` is read zero times in this codebase. The damage is analytical: registration-per-month reports silently under-count from May onward, and the numbers still look plausible, so nothing flags as broken.
- A separate ~5–6% background gap exists across 2023–Apr 2026, scattered evenly. Different cause, not established.

### Honest Accounting
- The `myatp` → no-stamp link is **inference**, not confirmation. It is strongly supported by the timeline and the attendance-token control case, but nobody has seen upstream's code. The PDF presents it as the cause; this was flagged to Fendy at the time.
- The report names commit `aa01c8b` and the exact file and line holding the token. Fine internally — worth softening if it goes to Integrasi, where the framing should be "which app identity gets stamped," not who committed what.

### Collaboration Highlights
- Fendy authorised writing to a real record (`1353151`) to test the update path, noting it could be erased afterwards — which moved the investigation past guesswork.
- Fendy pushed back on a report *page* and asked for plain words instead, then narrowed the ask twice ("just what's affected", "include the cause you mentioned"). The final deliverable was tighter for it.

### Looking Forward
- **Still bleeding.** Every registration since 17 April adds to the 15,819, and it is unfixed right now.
- Request a register-flow token issued to the **`alumnai`** application from Integrasi, matching what attendance already uses, and confirm with them that stamping is gated on app identity.
- Backfill is exact, not estimated: Firestore stores a `createTime` on every document, accurate to the second for all 15,819 affected records.
- Consider a locally-written `tarikh_register` so the app never again depends solely on an upstream field it cannot see fail.

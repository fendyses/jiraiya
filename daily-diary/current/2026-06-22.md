### What Was Fixed
- **Lite-mode black screen on first load** — The toggle persisted Lite via `localStorage`, so a fresh load *started* frozen. Freezing the render loop before the first frame paints = black canvas. Fix: dropped persistence — every load starts in Live (scene always paints), Lite is now a per-session toggle. Fendy confirmed the fix and committed it.

### Key Decisions & Discussion
1. **Dashboard declared "complete."** Walked through it honestly — twice asked "what do you think now." The engineering finally matches the visual ambition (dev-build smells gone, auto-pause, clean code). The tell that it's done: the "improvement" ideas started circling back to things Fendy had already deliberately cut (the agent stats card, then a mini-game). Both correctly rejected — discipline over feature-piling.
2. **No mini-game.** Considered, then dropped — a playable game turns a glance-able ambient cockpit into a task. Wrong for what the dashboard is.
3. **World selector — the first "what else" idea that genuinely fits.** A world = just the *backdrop* (layout + sky/day-night palette + ground + ambient FX); the agents/gauges/repos are the constant HUD.
4. **MP4-first hybrid (the wise call, Fendy's instinct).** Pre-rendered looping MP4s (day + night pairs, crossfaded by the real clock) for cheap, gorgeous variety — far easier than authoring more Three.js. The **3D village stays the one "living" world** where agents truly inhabit it; MP4 worlds are agent-free animated backdrops. Sketched the full architecture (video layer + selector + clock crossfade + Lite/visibility pause).
5. **Tools to animate the concept art:** image-to-video AI (Runway + Motion Brush for controlled motion, Kling, Sora, Luma, Veo; local via ComfyUI/Wan/SVD), then loop in DaVinci/CapCut and compress with ffmpeg.

### Key Insights & Learning
- **A JPEG/AI render can't be "made to move" — it's flat pixels, no geometry.** Real motion (water/wheel/lamps) needs either 3D geometry (what the village already does) or an MP4. The lush ChatGPT castle-island image is a *north star / concept art*, NOT a buildable or animatable asset — and even a perfect 3D rebuild won't match its baked lighting in real-time WebGL.
- **My real bottleneck in 3D isn't coding — it's blindness.** I place models by guessed coordinates; the current village got polished only because Fendy was my eyes via screenshots. The fix is a **screenshot harness**: a no-auth `preview.html` (just the 3D world) + a `shot.js` headless-capture script, so I render → screenshot → Read → adjust on my own. Levels above that (browser-automation MCP) exist but converge on the same senses (screenshots/console/DOM) — and Fendy's office experience confirmed live browser MCP is noticeably slow. The file-based harness is the lean version; defer building it until a real 3D world actually starts.
